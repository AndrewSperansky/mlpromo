## Что такое Staged Changes

### Staged Changes (или индекс) — это изменения, которые готовы для следующего коммита.

## В Git есть три состояния файла:

Состояние	Объяснение
1️⃣Modified	Файл изменён в рабочей директории, но Git ещё не знает, что с ним делать
2️⃣Staged	Файл добавлен в индекс (git add), готов к коммиту
3️⃣Committed	Изменение сохранено в истории Git

Пример:

## Изменили файл src/app.module.ts
git status
## Вывод: modified: src/app.module.ts

## Добавляем его в staged
git add src/app.module.ts
git status
## Теперь он в staged
## staged changes: src/app.module.ts

## Делаем коммит
git commit -m "feat: add new module"


## Как это выглядит в Git Graph

### Git Graph показывает три уровня:

1️⃣Working Directory / Modified — изменения в файлах, которые ещё не добавлены в индекс.
2️⃣Staged Changes — изменения, которые уже добавлены через git add.
3️⃣Commit History — уже сохранённые коммиты.

#### То есть staged — это промежуточный шаг между редактированием и коммитом.

## Почему staged нужен

## Позволяет выбрать, какие изменения пойдут в коммит, а какие пока нет.
## Можно коммитить частями, даже если несколько файлов изменены.

### Пример:

git add src/auth/auth.service.ts
git commit -m "feat: add login logic"

## Только этот файл будет в коммите, остальные остаются modified

## Удаление файла

git rm --cached src/.gitignore  # если он уже под Git
del src\.gitignore              # удалить физически

## Добавление файла

git add .gitignore
git commit -m "chore: remove duplicate .gitignore from src/"
git push origin main

# 🔹 Простыми словами

## Stash позволяет временно спрятать изменения в рабочей директории и staged файлах, чтобы переключиться на другую ветку или сделать срочный коммит, не теряя текущую работу.

## После этого рабочая директория возвращается к состоянию последнего коммита, будто ты ничего не менял.

🔹 Основные команды


| Действие                  | Команда                  | Пояснение                                                      |
| ------------------------- | ------------------------ | -------------------------------------------------------------- |
| Спрятать изменения        | `git stash`              | Сохраняет все изменения (staged и unstaged) во временный stash |
| Список stash              | `git stash list`         | Показать все сохранённые тайники                               |
| Применить stash           | `git stash apply`        | Восстанавливает изменения из stash, stash остаётся             |
| Применить и удалить stash | `git stash pop`          | Восстанавливает изменения и удаляет их из stash                |
| Удалить stash             | `git stash drop <stash>` | Удалить конкретный stash по имени                              |
| Очистить все stash        | `git stash clear`        | Удалить все тайники                                            |




## 🔹 Пример использования

## Ты работаешь над comments сервисом, но срочно нужно переключиться на main:

git stash          # прячем изменения
git checkout main  # переключаемся на main


## Решил срочную задачу на main, теперь возвращаемся к работе над сервисом:

git checkout feature/comments
git stash pop      # возвращаем изменения обратно


## ✅ Всё, что было спрятано, вернулось на место.



# наглядная мини-схема работы Git Stash — максимально простая и понятная:

┌────────────────────┐
│  Working Directory │   ← ты что-то изменил
└───────────┬────────┘
            │  git stash
            ▼
     ┌──────────────┐
     │    STASH     │  ← изменения спрятаны
     │ (тайник Git) │
     └──────┬───────┘
            │  git stash pop / apply
            ▼
┌────────────────────┐
│  Working Directory │  ← изменения возвращены
└────────────────────┘


## 🚀 Коротко логика процесса
### 1️⃣ Ты сделал изменения →

## Git видит modified / added файлы.

### 2️⃣ git stash →

## Изменения переносятся в stash
Рабочая директория становится «чистой», как будто только что сделал git pull.

### 3️⃣ Работаешь с ветками как нужно →

Переключаешься, делаешь фиксы, коммиты и т.п.

### 4️⃣ git stash pop →

Изменения возвращаются обратно
(и stash удаляется)


## 🔵 1. Что такое git rebase
### 📌 Смысл:

### Переписать историю своей ветки так, будто она была создана позже.
Т### ты как бы “перено́сишь” свои коммиты поверх актуальной версии другой ветки.

🔧 Пример: git rebase main
До rebase
(main) A──B──C
          \
(feature)  D──E


## После:

### После rebase feature на main

(main)    A──B──C
                   \
(feature new)       D'──E'


### D' и E’ — те же изменения, но как новые коммиты.


## Принудительно перезаписать удалённую ветку (правильный и безопасный)
git push -u origin main --force


## если хочешь сохранить тот один коммит с GitHub
git pull origin main --rebase
git push -u origin main

git fetch origin
git pull --rebase origin main
# решить конфликты если есть
git push origin main





🔵 2. Что такое HEAD
📌 Смысл:

HEAD — это указатель на текущий коммит, где ты сейчас работаешь.

Схема:
Коммиты:

A──B──C──D
        ▲
       HEAD  ← ты находишься на коммите D

Если переключишь ветку:
git checkout feature

A──B──C──D    ← ветка main
      \
       E──F   ← ветка feature
           ▲
          HEAD


HEAD просто переезжает туда, где ты работаешь.

🔵 3. Что такое origin
📌 Смысл:

origin — это имя удалённого репозитория, обычно GitHub.

Когда ты клонируешь проект:

GitHub repo  ← имя: origin
      │
      ▼
Local clone

Типичное отображение:
origin/main     — удалённая ветка на GitHub  
main            — твоя локальная ветка  
HEAD → main     — ты работаешь в локальной main

Когда ты делаешь push:
Локальная ветка main
            │ git push
            ▼
Удалённая ветка origin/main

# Cherry-pick
🧠 Что это такое

## git cherry-pick — это:

взять ОДИН конкретный коммит из одной ветки
и аккуратно применить его в другую ветку

Без:  
слияния веток
перетаскивания всего мусора
конфликтов “почему тут столько лишнего”


## 📦 Аналогия

Представь:

stage2/prod — показная версия

main — активная разработка

Ты:  
нашёл баг в prod  
починил его прямо в prod  
но хочешь, чтобы фикс был и в main  

➡️ Cherry-pick = “скопировать этот фикс туда”

5️⃣ Типовой сценарий cherry-pick (практика)
🔧 Шаг 1. Чиним баг в prod
git checkout stage2/prod

### правка кода
git commit -m "Fix JSON serialization for 1C payload"


Допустим, коммит получил хеш:

a1b2c3d

🍒 Шаг 2. Переносим фикс в main  
git checkout main  
git cherry-pick a1b2c3d  


Готово:
- тот же фикс
- новый коммит
- история чистая

📌 Ветки не склеились, ты перенёс ровно то, что нужно.

6️⃣ Когда cherry-pick — ПРАВИЛЬНО

✅ Hotfix из prod → main  
✅ Один логический фикс  
✅ Исправление бага  
✅ Мелкое улучшение

7️⃣ Когда cherry-pick — НЕ надо

❌ Массовый рефакторинг  
❌ 20 коммитов подряд  
❌ Слияние больших фич  

Для этого:

обычный merge  
или rebase (позже)

### если конфликт
git cherry-pick --abort
